# 웹(Web)이란 도대체 뭘까?

HTML/CSS/JS 다 아는데… 그래서 웹이 “어떻게” 동작하는 건데?

웹 공부를 한참 하다 보면 이런 생각이 든다.

> “그니까… HTML/CSS/JS는 알겠는데
> 
> 
> 그래서 ‘웹’이 뭐임? 왜 이게 화면이 되는 거지?”
> 

나도 그 타이밍에서 한참 멈춰 있었음.

그래서 오늘은 **웹의 동작 원리 관점**에서 전체를 정리해본다.

결론부터 말하면,

> 웹은 **“화면”이 아니라 “문서를 요청하고 렌더링하는 시스템”**이다.
> 

이걸 이해하면 HTML, CSS, JS가 왜 존재하는지까지 하나로 정리된다.

---

# 0. 웹의 본질: 문서를 “주소로 요청하고, 받으면 렌더링한다”

웹(Web)은 한 문장으로 요약된다.

> URL로 리소스를 식별하고 → HTTP로 요청/응답하고 → 브라우저가 해석해 보여준다.
> 
- **URL**
    
    “어디에 무엇이 있는지”를 나타내는 주소 체계
    
- **HTTP**
    
    그걸 “어떻게” 주고받을지 정하는 규칙
    
- **브라우저**
    
    받은 문서를 화면에 렌더링해주는 프로그램
    

여기서 HTML/CSS/JS는

브라우저가 해석하는 **문서(리소스)** 중 하나일 뿐이다.

---

# 1. HTML: “의미(Semantic)를 가진 문서”

### HTML이란?

**HTML (HyperText Markup Language)**

= “하이퍼텍스트 문서를 표현하는 마크업 언어”

여기서 중요한 건 “마크업 언어”라는 점.

- HTML은 기능·로직을 실행하는 언어가 아니라
- “이건 제목이다”, “이건 문단이다” 처럼
    
    **구조와 의미(semantic)** 를 표현하는 언어다.
    

그래서 HTML은 프로그래밍 언어가 아니다.

- 변수 없음
- 조건문 없음
- 반복문 없음
    
    → “이건 무엇이다”만 선언한다.
    

### 요소 / 속성

```html
<p>문단입니다.</p>
<imgsrc="cat.png"alt="고양이"width="200" />
```

HTML은 결국 **트리 구조(DOM)** 로 변환된다.

여기서부터 웹의 진짜 동작이 시작된다.

---

# 2. 브라우저는 HTML을 그대로 쓰지 않는다 → “DOM을 만든다”

HTML은 문자열일 뿐. 그대로 보여줄 수 없다.

그래서 브라우저는:

> HTML → DOM(Document Object Model)
> 

이 과정을 통해 JS에서 조작 가능한 객체 구조가 만들어진다.

- HTML = “문서”
- DOM = “문서를 객체처럼 다룰 수 있는 모델”

브라우저의 렌더링 파이프라인은

**DOM을 기반으로 돌아간다.**

---

# 3. CSS: 보이는 방식을 규칙으로 정하는 언어

CSS(Cascading Style Sheets)는 이렇게 정의된다.

> HTML 문서를 어떻게 보이게 할지 정하는 스타일 언어
> 

역할은 HTML과 완전히 다르다.

- HTML → 구조/의미
- CSS → 레이아웃·색·폰트·크기 등 시각적 표현

### CSS가 왜 등장했나?

옛날 HTML 시대엔 이런 식이었다.

- `<table>`로 레이아웃 만들고
- `<font>`로 폰트 바꾸고
- `bgcolor`로 배경색 지정하고…

결국 HTML이 문서가 아니라 **스타일링 스크립트**가 되어버렸다.

그래서 W3C가 선언한 것:

> “구조(HTML)와 표현(CSS)를 분리하자.”
> 

이게 지금 웹 개발의 근간이 됨.

### CSSOM

CSS도 결국 구조화된다.

> CSS → CSSOM(CSS Object Model)
> 

DOM & CSSOM 조합이 렌더링의 핵심.

---

# 4. JS(JavaScript): 동작과 로직을 담당하는 언어

JS는 “행동”을 만든다.

- HTML: 버튼이 있다
- CSS: 그 버튼의 디자인
- JS: 버튼을 누르면 모달이 등장한다

역할은 명확하다.

- 이벤트 처리(click, scroll, input)
- DOM 조작
- AJAX / fetch
- SPA(React/Vue)의 기반

웹 페이지가 생기면

“사용자와 상호작용”하는 모든 요소 뒤에 JS가 있다.

---

# 5. 브라우저 렌더링 파이프라인:

HTML/CSS/JS가 “화면”이 되는 과정

웹을 깊이 이해하려면 이 부분이 핵심이다.

브라우저는 아래 순서로 일한다.

1. HTML 다운로드
2. HTML 파싱 → **DOM 생성**
3. CSS 다운로드
4. CSS 파싱 → **CSSOM 생성**
5. DOM + CSSOM → **Render Tree 생성**
6. **Layout(Reflow)**: 각 요소의 위치/크기 계산
7. **Paint**: 픽셀로 그리기
8. **Composite**: 여러 레이어 합쳐서 화면 출력

> 화면은 DOM + CSSOM을 기반으로 렌더링된다.
> 

JS가 DOM/CSSOM을 조작하면?

→ Layout, Paint가 다시 발생 → 버벅임 발생 가능

그래서 “리플로우 최소화”, “쓰로틀/디바운스”, “requestAnimationFrame” 같은 단어들이 등장한다.

---

# 6. DOM & CSSOM & JS 관계 정리

- **HTML → DOM**
- **CSS → CSSOM**
- **JS → DOM/CSSOM 조작 + 이벤트 처리 + 서버 통신**

이 세 가지가 웹의 본체라고 보면 된다.

---

# 7. JS는 어디서 실행될까?

엔진(V8)과 브라우저 Web APIs의 역할

JS 자체는 **브라우저 엔진**에서 실행된다.

- Chrome → V8
- Firefox → SpiderMonkey
- Safari → JavaScriptCore

근데 중요한 사실:

> JS 엔진은 “순수 JS 코드만” 실행한다.
> 
> 
> DOM 조작 / AJAX / setTimeout 같은 건 JS가 아니라 **브라우저 Web API**가 제공한다.
> 

JavaScript는 브라우저 생태계의 일부일 뿐.

- JS 엔진: 코드 실행
- Web API: DOM, 네트워크, 타이머 등 기능 제공
- 이벤트 루프: 둘 사이를 연결

---

# 8. 내가 이해한 “웹”의 정의

정리하자면,

웹은 단순히 “HTML + CSS + JS”가 아니다.

> 웹은 URL로 리소스를 요청하고,
브라우저가 DOM/CSSOM으로 해석해 렌더링하는 거대한 시스템이다.
> 

그리고 그 위에서 우리가 웹앱, SPA, 프레임워크(React/Vue)를 올리는 것.

---

# 마치며

웹은 “기술 모음”이 아니라

**문서/자원 요청 → 해석 → 렌더링**이라는

하나의 거대한 파이프라인이다.

이 관점에서 보면 HTML/CSS/JS가 왜 그렇게 설계되었는지, React/Vue가 왜 필요한지, 웹 성능이 왜 중요한지까지 깔끔하게 이어진다.
